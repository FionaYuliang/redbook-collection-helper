# 设计思路

本文档统一记录“小红书收藏整理助手”的产品与技术设计。每次修改功能、信息架构、数据模型、界面结构或关键交互后，都需要同步更新本文档，避免设计依据只散落在提交记录或对话中。

## 设计原则

- 本地优先：用户收藏、分类、主题、收藏夹和设置默认保存在浏览器本地 IndexedDB。
- 轻量导入：MVP 通过内容脚本扫描当前小红书页面 DOM，不依赖私有接口，不修改小红书数据。
- 渐进增强：未配置 AI 时使用本地规则分类；配置 OpenAI-compatible Chat Completions 后，只把必要的笔记元数据发送给外部模型。
- 可回退：AI 分类失败、页面结构变化或导入中断时，已导入数据仍然可搜索、筛选和手动整理。
- 侧边栏工作流：主体验集中在 Chrome Side Panel，适合边浏览边导入、边整理。

## 产品结构

当前 MVP 面向“把收藏变成可整理资料库”的最短路径：

- 导入：从当前小红书页面真实发出的收藏相关 Web 接口中捕获请求，再用同一登录态分页读取收藏笔记；不再默认使用读屏滚动采集。
- 统计：在顶部展示总收藏、一级分类、子分类数量；远端总收藏优先使用页面里的“笔记”计数，避免被个人主页或导航里的“收藏”计数覆盖。
- 分类：用最多 3 级的逐层分类页面组织笔记；当前分类有下级时进入下一层分类页，没有下级时进入笔记列表。
- 收藏：笔记列表是分类路径的最底层页面，只提供搜索，默认按收藏时间从近到远排序。
- 收藏夹：允许创建项目型收藏夹，并把笔记加入收藏夹。
- 设置：配置 AI endpoint、模型名和 API Key。

## 信息架构

Side Panel 分为四个主标签：

- 分类：默认视图，按一级、二级、三级分类逐层展示，是整理入口。
- 收藏：作为最底层笔记列表出现，只保留搜索、移动分类和详情查看。
- 收藏夹：面向项目型收集，把跨分类笔记聚合到用户自定义集合。
- 设置：面向 AI 分类能力和本地配置。

默认一级分类覆盖高频收藏场景：旅游、家居、育儿、学习成长、数码、美食、健康运动、宠物、汽车、其他。用户可在 1-2 级分类下新增子分类，笔记可移动到任意一级到三级分类。

## 数据设计

IndexedDB 数据库名：`xhs_collection_assistant`，当前版本：`1`。

核心对象仓库：

- `notes`：笔记主数据，包含标题、作者、封面、原始链接、收藏时间、导入时间、分类、更新时间、删除标记等。
- `categories`：树形分类，包含名称、父分类、层级、排序、系统分类标记和时间戳，最多 3 级。
- `topics`：旧二级主题兼容数据，归属于某个分类。
- `note_topics`：旧主题关系兼容数据。
- `collections`：用户创建的项目型收藏夹。
- `collection_notes`：收藏夹和笔记的多对多关系。
- `meta`：同步元数据，例如最近导入时间。
- `settings`：AI endpoint、模型名、API Key 等本地设置。

设计取舍：

- 分类字段保留在 `notes` 上，便于按当前分类路径进入笔记列表；父分类的笔记列表包含其全部子分类笔记。
- 旧二级主题关系保留作兼容，当前主要整理入口为树形分类。
- 收藏夹与分类体系分离，避免项目型整理破坏内容分类。

## 导入与同步

内容脚本负责从小红书页面提取笔记：

- 从 `a[href]` 中识别笔记 ID，并优先保存小红书页面暴露的完整原始链接，保留 `xsec_token`、`xsec_source` 等访问参数；缺失有效链接时再回退到 `https://www.xiaohongshu.com/explore/{id}`。
- 从卡片附近 DOM 读取标题、作者、封面、可识别的官方收藏时间和页面来源；收藏时间只在页面 DOM 或相关属性中暴露时可获取，不使用导入时间伪装。
- 通过有限滚动收集更多可见卡片，连续多轮无新增后停止。
- 用笔记 ID 去重，重复导入时更新已有记录而不是生成副本。
- 设置页提供“清空并重新导入”，会清空本地笔记、分类、主题、收藏夹和同步统计，保留设置后从当前小红书页面重新扫描。
- 同步设置允许调整最大采集触发次数。若没有捕获到收藏接口，会提示先在小红书页面打开收藏列表后重试。
- 接口不可用时必须询问用户是否回退到页面采集；页面采集期间 loading 面板显示“停止采集”按钮，用户可中断扫描。

Side Panel 负责触发导入、写入 IndexedDB、刷新本地状态和展示同步结果。导入只处理当前页面能扫描到的数据；如果小红书页面结构变化，需要优先调整 `src/content-script.js` 的提取策略。

## 分类策略

分类采用“用户意图优先”的多级兜底：

- 用户创建的大类优先：已有一级分类是分类边界，笔记应尽量进入用户定义的大类。
- 笔记 tag 优先：如果小红书话题/tag 能明确匹配用户大类或大类关键词，直接归入该大类，不让 AI 覆盖。
- AI 分类：用户配置 API Key 后，调用 OpenAI-compatible Chat Completions 接口批量分类剩余不确定笔记。
- 本地规则兜底：AI 不可用时，根据标题、作者、原始专辑、tag 和大类/主题关键词匹配。
- “其他”救援：初次分类后若“其他”超过 30%，自动对“其他”笔记重新读取关键词并二次归类；若剩余笔记中出现高频有效 tag，会自动创建新的一级分类承接这一组，否则保留在“其他”。

隐私边界：

- API Key 只保存在本地 IndexedDB。
- 外部 AI 请求发送笔记 `id`、标题、作者、原始专辑和可提取到的原笔记 `tags` 字段；分类时优先参考 tag。
- 不上传用户账号、Cookie、完整页面内容或自有服务器数据。

## 界面设计

视觉方向是轻量、温暖、工具型：

- 使用浅色背景、低对比边框和 8px 圆角，保持侧边栏里的信息密度。
- 主色使用偏小红书语境的红色作为导入和主操作强调。
- 导航和工具栏尽量稳定，减少整理过程中的跳动。
- 卡片用于重复列表项，不把页面区域层层包成卡片。
- 所有关键操作需要有即时反馈，例如 toast、统计数字更新和列表刷新。
- 导入期间侧边栏进入锁定态，显示“准备中 / 采集中 / 导入中 / 整理中 / 完成中”等阶段，避免用户操作打断滚动采集或分类写库流程。
- 开始导入前必须确认当前活动页面是小红书收藏页，URL 应为 `/user/profile/...?...tab=fav`；采集中显示 `当前采集数量/总收藏笔记数量` 进度。

## 技术约束

- Chrome Extension Manifest V3。
- 无构建步骤，源码直接由扩展加载。
- 校验入口是 `npm run check`，负责检查 manifest、必要文件和 JavaScript 语法。
- 图标源文件为 SVG，PNG 图标由 `tools/make-icons.py` 生成。

## 后续设计方向

- 更可靠的导入：针对小红书页面结构变化补充多选择器策略和失败提示。
- 更强的整理：支持批量移动分类、批量加收藏夹、未整理队列。
- 更透明的 AI：展示分类置信度、分类原因和可一键修正入口。
- 更稳的数据演进：新增 IndexedDB 版本迁移文档和数据导出/导入能力。
